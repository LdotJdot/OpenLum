OpenLum — REPL 案例与「大哥级」效率对标（可复现）

说明
- 「大哥」= 少轮次、少 token、选对工具、不滥用子会话。以下用脚本可回归对比。
- 整段多行一条消息：--repl-file <path> --repl-file-entire
- 多轮一行一条：--repl-file <path>（每行一轮，末尾 /quit）

案例文件（整文件单轮，建议用 --repl-file-entire）
- entire-01-read-not-list.example.txt
  目标：给定具体文件路径时优先 read，不要整目录 list_dir。
- entire-02-grep-then-read.example.txt
  目标：先 grep 定位，再 read 小窗口，避免整文件读。
- entire-03-listdir-glob.example.txt
  目标：浅层 list_dir + name_glob 过滤，不扫全目录。
- entire-04-readonly-batch.example.txt
  目标：一轮内 grep + read_many（或并行 read），无 sessions_spawn。
- entire-05-workflow-exec.example.txt
  目标：需写/执行时：submit_plan → exec（或 write），符合 workflow。

- entire-11-read-docx-sample-math.example.txt

- entire-12-fly-search-craes.example.txt / entire-13-fly-multi-search.example.txt

    --repl-file OpenLum.Console\e2e-repl-examples\entire-01-read-not-list.example.txt --repl-file-entire

准确率：对照上表「参考」人工扫 assistant 最终回复；效率：同一会话 ToolRounds/ModelTurns 越少越好（无硬性上限，用于对比迭代）。

系统提示里已增加 **## Efficiency (tokens & turns)**（抽象规则，与具体工具 description 一致）。
工具层已有：list_dir name_glob、read 文案、grep/glob 状态行说明、sessions_spawn 描述等。

配置侧（openlum.json）
- compaction.maxToolResultChars / maxFailedToolResultChars：控制进历史的工具结果长度。
- tools.deny：可去掉不用的浏览器或 sessions_spawn 等以缩小 schema、省 token。
- conversation.autoSave + path：每轮用户对话结束后自动写入 .openlum（相对 workspace）。
- 环境变量 OPENLUM_CONVERSATION_FILE：覆盖自动保存路径。
- 续聊：openlum-console.exe --continue path.openlum，或 openlum-console.exe path.openlum（简写）。
- promptOverlay：可选字符串，追加在固定系统提示之后（## Config overlay），用于部署/团队约束；不写具体工具名，避免与 **Tools** 重复；工具用法仍以各工具 description 为准。

workflow（openlum.json）
- requirePlanForWrite：为 true 时，写文件类工具需先 submit_plan 或 TODO 进入 Act；与 exec 无关（见 allowExecInObserve）。
- allowExecInObserve：默认 true，Observe 阶段即可 exec（少一轮规划）；设 false 则恢复「exec 需进 Act」的旧行为。

与 Cursor 类环境类似的层次
- 固定系统提示：角色、流程、安全、抽象效率原则（OpenLum：SystemPromptBuilder）。
- 工具层：名称 + 参数 + 怎么用（各 ITool.Description）。
- 任务层：用户一句话 / 脚本 / Skill（领域步骤），不要把领域步骤写进固定系统提示。
- 可选 promptOverlay：统一注入团队规范（仍建议保持抽象）。
